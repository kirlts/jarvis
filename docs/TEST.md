## Testing Stack

| Layer | Runner / Tool | Version | Justification |
|---|---|---|---|
| **1. Contract Testing** | Specmatic | latest | OpenAPI spec is the single source of truth for core↔plugin contracts. Specmatic generates consumer mocks and validates provider compliance automatically. Replaces Pact (sequential workflow incompatible with solo-dev). |
| **2. Mutation Testing** | Stryker | latest | Auditor of AI-generated tests. Introduces deliberate bugs (mutants) and verifies test suite detects them. Surviving mutants = mathematical proof of coverage gaps. |
| **3. Integration Testing** | Testcontainers (Node.js) | ^11.14.0 | Spins up PG 17 + PgBouncer + MinIO as real Docker containers inside test suites. No mocks for infrastructure. Already installed as devDependency. |
| **4. Property-Based Testing** | fast-check | latest | Generates thousands of random inputs to validate invariants (RLS isolation, UUIDv7 ordering, payload schema boundaries). Finds edge cases that example-based tests miss. |
| **5. Stress Testing** | K6 + xk6-dashboard | latest | Go-based load generator. 30K-40K VUs from single machine. xk6-dashboard for real-time visualization without Grafana dependency during test execution. |
| **6. Unit Testing** | node:test (built-in) | Node 24.x | Native test runner, zero dependencies. Used ONLY for pure functions (schema validation, UUID generation, config parsing). Mocking infrastructure is prohibited at this layer. |

### Runner Configuration

```
# Primary runner (unit + integration)
node --test src/**/*.test.js

# Specmatic (contract validation)
specmatic test --host localhost --port 3000

# Stryker (mutation audit)
npx stryker run

# K6 (stress scenarios)
k6 run --out dashboard scripts/stress/*.js
```

### What This Stack Does NOT Include

| Excluded | Reason |
|---|---|
| **Pact** | Sequential consumer-first workflow is an anti-pattern for solo-dev with AI assistance. Specmatic inverts this by using OpenAPI spec as source of truth. |
| **Jest** | CJS-first, requires transforms for ESM. `node:test` is native to Node 24 and zero-config. |
| **Playwright / E2E** | Jarvis is a headless API backend. No browser UI to test. Appsmith (Ops Console) is a third-party tool; its UI is not our testing responsibility. |
| **Artillery** | Single-threaded, commercial for distributed load. K6 is Go-native, free, and supports 30K+ VUs from a laptop. |

## Inverted Testing Pyramid

The traditional pyramid (many unit tests → few integration → fewer E2E) is **discarded**. For an AI-accelerated solo-dev building a multi-tenant SaaS:

```
         ┌─────────────────────┐
         │   Contract Tests    │  ← Specmatic: inviolable boundaries
         │   (OpenAPI spec)    │     core↔plugin, tenant↔admin
         ├─────────────────────┤
         │  Mutation Tests     │  ← Stryker: audits AI-generated tests
         │  (Stryker)          │     surviving mutants = test gaps
         ├─────────────────────┤
         │  Integration Tests  │  ← Testcontainers: real PG, real MinIO
         │  (Testcontainers)   │     no mocks, no stubs for infra
         ├─────────────────────┤
         │  Property-Based     │  ← fast-check: invariant validation
         │  (fast-check)       │     RLS, UUIDv7, schema boundaries
         ├─────────────────────┤
         │  Stress Tests       │  ← K6: physical limits validation
         │  (K6 + dashboard)   │     OOM, EMFILE, connection pools
         ├─────────────────────┤
         │  Unit Tests         │  ← node:test: pure functions ONLY
         │  (node:test)        │     no mocks of infrastructure
         └─────────────────────┘
```

**Rationale:** 
- **Mutation vs Mocks:** Unit tests with mocks create a false sense of security. They validate internal wiring, not system behavior. When the AI generates tests, it generates mocks that mirror the implementation, producing circular validation. Mutation testing (Stryker) breaks this cycle by proving whether the test suite can detect real bugs, independent of implementation knowledge.
- **Contract Testing vs Operaciones:** Contract tests (Specmatic) reemplazan los frágiles tests E2E validando la frontera del Core (Fastify) directamente contra las especificaciones OpenAPI. Esto garantiza que futuras actualizaciones o refactorizaciones internas del Core nunca rompan silenciosamente a los consumidores externos (Ops Console, Plugins) en producción.

## OpenAPI Spec Requirements

Specmatic requires an OpenAPI 3.x spec as its source of truth. The following specs must be created and maintained as first-class artifacts:

| Spec | Covers | Location |
|---|---|---|
| `specs/tenant-api.yaml` | `/api/v1/sync/inbox`, `/api/v1/storage/presign`, `/health` | To be created |
| `specs/admin-api.yaml` | `/admin/tenants`, `/admin/jobs`, `/admin/whatsapp/status` | To be created |

**Maintenance Guarantees:**
These specs are the **contracts** between the Core and all consumers (Appsmith, Plugins, WhatsApp workers). Specmatic garantiza facilidad de mantención futura sin afectar operaciones mediante:
1. **Single Source of Truth:** Valida que Fastify cumpla estrictamente el OpenAPI vía peticiones HTTP reales (sin mocks frágiles).
2. **Backward-Compatibility:** Cualquier refactor del Core que inadvertidamente elimine un campo esperado o altere un código HTTP esperado fallará el build local antes del pase a producción.
3. **Desacoplamiento Operativo:** Permite evolucionar el backend independientemente del frontend (Ops Console), asegurando matemáticamente que el Core inmutable nunca colapse las integraciones de los inquilinos.

## Workflow: AI-Generated Test Governance

```
1. AI generates test file (*.test.js)
2. AI runs tests → all pass
3. Stryker mutates the source code under test
4. Stryker re-runs the test suite against each mutant
5. Report: which mutants survived?
   → Surviving mutant = the test suite has a hole
   → AI generates additional tests to kill survivors
6. Repeat until mutation score > 80%
7. Human reviews Stryker report, not individual tests
```

This workflow ensures that AI-generated tests are not circular validations of the implementation. The human reviews the mutation report, not the test code, because the mutation report is the mathematical proof of test quality.
